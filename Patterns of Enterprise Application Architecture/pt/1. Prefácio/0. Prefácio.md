Na primavera de 1999, voei para Chicago para prestar consultoria em um projeto que estava sendo realizado pela ThoughtWorks, uma empresa de desenvolvimento de aplicações pequena, mas em rápido crescimento. O projeto era um daqueles ambiciosos projetos de aplicações empresariais: um sistema de retaguarda para leasing. Essencialmente, ele lida com tudo o que acontece com um contrato de leasing depois que você assina na linha pontilhada: enviar contas, lidar com alguém que atualiza um dos ativos do contrato, perseguir pessoas que não pagam suas contas em dia e descobrir o que acontece quando alguém devolve os ativos antecipadamente. Isso não parece tão ruim até você perceber que os contratos de leasing são infinitamente variados e terrivelmente complicados. A "lógica" de negócio raramente se encaixa em qualquer padrão lógico, porque, afinal, ela é escrita por pessoas de negócios para conquistar negócios, nos quais pequenas variações incomuns podem fazer toda a diferença para ganhar um acordo. Cada uma dessas pequenas vitórias acrescenta ainda mais complexidade ao sistema.

Esse é o tipo de coisa que me deixa empolgado: como pegar toda essa complexidade e criar um sistema de objetos que possa tornar o problema mais tratável. De fato, acredito que o principal benefício dos objetos está em tornar uma lógica complexa tratável. Desenvolver um bom Modelo de Domínio para um problema de negócio complexo é difícil, mas maravilhosamente satisfatório.

Ainda assim, esse não era o fim do problema. Nosso modelo de domínio precisava ser persistido em um banco de dados e, como muitos projetos, estávamos usando um banco de dados relacional. Também precisávamos conectar esse modelo a uma interface de usuário, fornecer suporte para permitir que aplicações remotas usassem nosso software e integrar nosso software com pacotes de terceiros. Tudo isso em uma nova tecnologia chamada J2EE, que ninguém no mundo tinha qualquer experiência real em usar.

Embora essa tecnologia fosse nova, tínhamos o benefício da experiência. Eu vinha fazendo esse tipo de coisa havia muito tempo com C++, Smalltalk e CORBA. Muitos dos profissionais da ThoughtWorks tinham muita experiência com Forte. Já tínhamos as principais ideias arquiteturais em nossas cabeças e só precisávamos descobrir como aplicá-las ao J2EE. Olhando para isso três anos depois, o design não é perfeito, mas resistiu muito bem ao teste do tempo.

Esse é o tipo de situação para a qual este livro foi escrito. Ao longo dos anos, vi muitos projetos de aplicações empresariais. Esses projetos frequentemente contêm ideias de design semelhantes que se mostraram eficazes para lidar com a complexidade inevitável que as aplicações empresariais possuem. Este livro é um ponto de partida para registrar essas ideias de design como padrões.

O livro está organizado em duas partes, sendo a primeira parte um conjunto de capítulos narrativos sobre vários tópicos importantes no design de aplicações empresariais. Esses capítulos apresentam vários problemas na arquitetura de aplicações empresariais e suas soluções. No entanto, eles não entram em muitos detalhes sobre essas soluções. Os detalhes das soluções estão na segunda parte, organizados como padrões. Esses padrões são uma referência, e não espero que você os leia de capa a capa. Minha intenção é que você leia os capítulos narrativos da Parte 1 do início ao fim para obter uma visão ampla do que o livro aborda; depois, consulte os capítulos de padrões da Parte 2 conforme seu interesse e suas necessidades o conduzirem. Assim, o livro é um livro narrativo curto e um livro de referência mais longo combinados em um só.

Este é um livro sobre design de aplicações empresariais. Aplicações empresariais tratam da exibição, manipulação e armazenamento de grandes quantidades de dados frequentemente complexos e do suporte ou automação de processos de negócio com esses dados. Exemplos incluem sistemas de reservas, sistemas financeiros, sistemas de cadeia de suprimentos e muitos outros que administram os negócios modernos. Aplicações empresariais têm seus próprios desafios e soluções particulares e são diferentes de sistemas embarcados, sistemas de controle, telecomunicações ou softwares de produtividade para desktop. Portanto, se você trabalha nessas outras áreas, não há realmente nada neste livro para você, a menos que queira ter uma ideia de como são as aplicações empresariais. Para um livro geral sobre arquitetura de software, eu recomendaria POSA.

Há muitas questões arquiteturais na construção de aplicações empresariais. Receio que este livro não possa ser um guia abrangente para elas. Na construção de software, acredito muito no desenvolvimento iterativo. No coração do desenvolvimento iterativo está a noção de que você deve entregar software assim que tiver algo útil para o usuário, mesmo que não esteja completo. Embora existam muitas diferenças entre escrever um livro e escrever software, essa é uma noção que acredito que os dois compartilham. Dito isso, este livro é um compêndio incompleto, mas, confio eu, útil de conselhos sobre arquitetura de aplicações empresariais. Os principais tópicos sobre os quais falo são

  - Divisão das aplicações empresariais em camadas
  - Estruturação da lógica de domínio, ou de negócio
  - Estruturação de uma interface de usuário Web
  - Ligação de módulos em memória, particularmente objetos, a um banco de dados relacional
  - Tratamento do estado de sessão em ambientes sem estado
  - Princípios de distribuição

A lista de coisas sobre as quais não falo é bem mais longa. Eu realmente gostaria de escrever sobre organização da validação, incorporação de mensageria e comunicação assíncrona, segurança, tratamento de erros, clustering, integração de aplicações, refatoração arquitetural, estruturação de interfaces de usuário para clientes ricos, entre outros tópicos. No entanto, devido a restrições de espaço e tempo e à falta de reflexão, você não os encontrará neste livro. Só posso esperar ver alguns padrões para esse trabalho em um futuro próximo. Talvez eu faça um segundo volume algum dia e entre nesses tópicos, ou talvez outra pessoa preencha essas e outras lacunas.

Entre esses assuntos, a comunicação baseada em mensagens é uma questão particularmente importante. Pessoas que estão integrando várias aplicações estão fazendo uso cada vez maior de abordagens de comunicação assíncrona baseada em mensagens. Há muito a ser dito sobre usá-las também dentro de uma aplicação.

Este livro não pretende ser específico para nenhuma plataforma de software em particular. Encontrei esses padrões pela primeira vez enquanto trabalhava com Smalltalk, C++ e CORBA no final dos anos 80 e início dos anos 90. No final dos anos 90, comecei a trabalhar extensivamente com Java e descobri que esses padrões se aplicavam bem tanto aos primeiros sistemas Java/CORBA quanto aos trabalhos posteriores baseados em J2EE. Mais recentemente, venho fazendo alguns trabalhos iniciais com a plataforma .NET da Microsoft e descobri que os padrões se aplicam novamente. Meus colegas da ThoughtWorks também acrescentaram suas experiências, particularmente com Forte. Não posso afirmar que exista generalidade entre todas as plataformas que já foram ou serão usadas para aplicações empresariais, mas até agora esses padrões demonstraram recorrência suficiente para serem úteis.

Forneci exemplos de código para a maioria dos padrões. Minha escolha de linguagem para eles é baseada no que acredito que a maioria dos leitores provavelmente será capaz de ler e compreender. Java é uma boa escolha nesse caso. Qualquer pessoa que consiga ler C ou C++ consegue ler Java, mas Java é muito menos complexo que C++. Essencialmente, a maioria dos programadores de C++ consegue ler Java, mas não o contrário. Sou um fanático por objetos, então inevitavelmente me inclino para uma linguagem orientada a objetos. Como resultado, a maioria dos exemplos de código está em Java. Enquanto eu trabalhava no livro, a Microsoft começou a estabilizar seu ambiente .NET, e sua linguagem C# tem a maioria das mesmas propriedades que Java para um autor. Então também fiz alguns exemplos de código em C#, embora isso tenha introduzido algum risco, pois os desenvolvedores não têm muita experiência com .NET e, portanto, as formas idiomáticas de usá-lo bem são menos maduras. Ambas são linguagens baseadas em C, então, se você consegue ler uma, deve conseguir ler ambas, mesmo que não esteja profundamente envolvido com essa linguagem ou plataforma. Meu objetivo era usar uma linguagem que a maior quantidade de desenvolvedores de software pudesse ler, mesmo que ela não fosse sua linguagem principal ou preferida. Minhas desculpas àqueles que gostam de Smalltalk, Delphi, Visual Basic, Perl, Python, Ruby, COBOL ou qualquer outra linguagem. Sei que vocês acham que conhecem uma linguagem melhor que Java ou C#. Tudo o que posso dizer é que eu também conheço!

Os exemplos estão ali para inspiração e explicação das ideias presentes nos padrões. Eles não são soluções prontas; em todos os casos, você precisará fazer uma boa quantidade de trabalho para adaptá-los à sua aplicação. Padrões são pontos de partida úteis, mas não são destinos.
